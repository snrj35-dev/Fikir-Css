# Changelog Policy

## Source of truth
Release notes under `docs/release/` are the authoritative changelog source.

## Minimum sections per release note
- Summary
- Highlights
- Behavioral Notes
- Upgrade Notes
- Release Checklist reference

## Writing rules
- Mention concrete canonical surfaces affected
- Call out breaking/behavioral changes explicitly
- Link migration notes when migration is needed
- Avoid vague entries like "improvements" without scope

## Lifecycle
- Create draft note during implementation
- Finalize note before tagging release
- Do not tag release without release note entry
