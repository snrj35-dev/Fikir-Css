# v1.0 Migration Guarantee Policy

> Created: 2026-04-12
> Scope: M4 — stability commitments for v1.0 consumers

## Guarantee Scope

These guarantees apply only to surfaces with `supported` status in `docs/roadmap/support-matrix.md` as of `v1.0.0`.

`experimental` surfaces are **not covered** — they may change without notice.

## Semver Commitment

| Type | Rule |
|------|------|
| Patch (x.y.Z) | Bug fixes only. No selector changes, no token renames. |
| Minor (x.Y.0) | New supported surfaces, new tokens. Existing selectors and tokens unchanged. |
| Major (X.0.0) | Breaking changes allowed with migration guide. |

## What is Frozen at v1.0

1. **Selector names** — All selector base names in `contracts/naming.contract.mjs` for `supported` surfaces.
2. **Token names** — All CSS custom property names in `packages/tokens/core.css` and `packages/tokens/semantic.css`.
3. **Data-attribute contracts** — All `data-*` attributes defined in `docs/architecture/headless-contract-spec.md`.
4. **Recipe API** — All recipe keys and variant axes in `contracts/recipes.contract.mjs`.
5. **Layer order** — `reset → base → layouts → recipes → components → utilities`.

## What is NOT Frozen

1. CSS property values (e.g., a `--space-4` value may change in minor releases if justified).
2. Internal implementation details within a CSS rule.
3. `experimental` surfaces.
4. Scripts and tooling in `scripts/`.
5. Documentation structure.

## Deprecation Window

When a `supported` surface or token must be changed in a breaking way:
1. Deprecation notice added to `dist/contracts/alias-migration.json`
2. Old selector remains functional for **2 minor versions**
3. Migration guide published in `docs/migration/`
4. Breaking change released in the next major version

See `docs/contracts/selector-deprecation-window-policy.md` for the full deprecation procedure.
