# v0.4 Promotion Candidates (Experimental -> Supported)

> Last reviewed: 2026-04-11

## Purpose
Define the first candidate set for v0.4 support-level promotion workstream.

Promotion criteria source:
- `docs/release/experimental-to-supported-criteria.md`

## Candidate Set (P1)
1. `textarea`
- Reason: core form workflows need multiline input as supported baseline.
- Current state: implemented + tested + documented examples.

2. `select`
- Reason: baseline form control expected as supported in product-ready UI kits.
- Current state: implemented + tested + documented examples.

3. `checkbox`, `radio`, `switch`
- Reason: choice controls are required for realistic settings/forms.
- Current state: implemented + tested + documented examples.

4. Overlay candidate: `toast`
- Reason: lightweight feedback pattern with strong practical usage.
- Current state: implemented + tested + playground interactions.

5. Navigation/data candidate: `pagination`
- Reason: common list/table workflow requirement.
- Current state: implemented + tested + demo examples.

## Promotion Waves
- Wave 1: `textarea`, `select`, `checkbox/radio/switch`
- Wave 2: `toast`, `pagination`

## Required Evidence Checklist per Candidate
- RFC/spec link
- implementation evidence (`packages/components/*.css`)
- test evidence (`tests/build` / `tests/source`)
- accessibility note/checklist references
- playground/docs usage evidence
- release/migration impact review

## Out of Scope for This Candidate List
- New components (`number-input`, `rating`, `result`) are separate gap-closure workstream tasks.
- Advanced surfaces (`tree-view`, `tags-input`) remain P2.
